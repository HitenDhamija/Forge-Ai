# Architecture

## Tech Stack

- **Node.js**: Version `22.12.0` as specified in `package.json`.
- **Express**: Version `5.1.0` used for building RESTful APIs.
- **MongoDB**: Version not explicitly stated, but MongoDB is referenced through the use of Mongoose version `8.16.1`.
- **Cloudinary**: For handling image uploads and storage with Cloudinary configuration in `cloudConfig.js`.
- **dotenv**: Used to load environment variables from `.env` files.
- **ejs-mate**: Template engine for rendering EJS templates.
- **method-override**: Middleware for supporting PUT, DELETE requests via the HTTP verbs 'X-HTTP-Method-Override' and 'X-HTTP-METHOD-OVERRIDE'.
- **joi**: For schema validation in `schema.js`.
- **mongoose**: Version `8.16.1` used to interact with MongoDB.
- **multer**: Middleware for handling file uploads, specifically `multer-storage-cloudinary` version `4.0.0` is utilized for uploading images to Cloudinary.
- **passport**: For authentication and authorization middleware, including `passport-local` and `passport-local-mongoose`.
- **connect-flash**: Used for flash messages in Express applications.

## Project Structure

The project structure follows a typical MVC (Model-View-Controller) pattern:

- **app.js**: Entry point of the application.
- **controllers/**: Contains all controller files which handle HTTP requests. Each file corresponds to an API endpoint and is responsible for handling CRUD operations.
  - `listings.js`: Handles listing-related endpoints.
  - `reviews.js`: Handles review-related endpoints.
  - `users.js`: Handles user-related endpoints.
- **middleware/**: Contains middleware functions that are used to handle common tasks such as authentication, error handling, etc.
  - `isLoggedIn.js`: Middleware for checking if a user is logged in before allowing them to perform certain actions.
  - `saveredirectUrl.js`: Middleware for saving the redirect URL from session and setting it in locals.
- **models/**: Contains mongoose schemas/models for interacting with MongoDB. Each model corresponds to a database collection.
  - `listing.js`: Represents the schema for listings.
  - `review.js`: Represents the schema for reviews.
  - `user.js`: Represents the schema for users.
- **package.json**: Application configuration file, including dependencies and scripts.
- **public/**: Contains static files such as CSS and JavaScript files used by EJS templates.
- **routes/**: Contains route definitions. Each file corresponds to a specific API endpoint.
  - `listing.js`: Defines routes related to listings.
  - `review.js`: Defines routes related to reviews.
  - `user.js`: Defines routes related to users.
- **schema.js**: Contains schema validation functions using Joi for mongoose schemas.
- **utils/**: Contains utility files such as error handling and middleware.
  - `ExpressError.js`: Utility file for creating custom Express errors.

## Data Flow

### Request Lifecycle
1. **Client-Side**:
   - Users interact with the application through HTTP requests to API endpoints defined in routes.
2. **Middleware Handling**:
   - Middleware functions are executed first, such as checking authentication (`isLoggedIn`), saving redirect URLs (`saveredirectUrl`).
3. **Controller Handling**:
   - Controller functions handle business logic and interact with models or databases (e.g., `Listing.find`, `Review.create`, etc.).
4. **Model Interaction**:
   - Models are used to query, create, update, and delete data from MongoDB.
5. **Response Generation**:
   - Controllers generate responses based on the results of model interactions.
6. **Client-Side Rendering**:
   - EJS templates are rendered using `ejs-mate`, with locals populated by middleware functions like `saveredirectUrl`.
7. **Static Files Serving**:
   - Static files from the public directory are served directly to clients.

### Data Flow Diagram
- Users interact via HTTP requests.
- Middleware handles authentication and redirects.
- Controllers handle business logic, interacting with models for data retrieval and manipulation.
- Models interact with MongoDB databases.
- Responses are generated by controllers and rendered using EJS templates.
- Static files from the public directory are served directly.

## Key Components

### Main Modules/Classes
1. **Listing**: Represents a listing in the database. Contains properties like title, description, location, price, country, images, etc., with validation through `listingSchema`.
2. **Review**: Represents a review associated with a listing. Includes author information and other details.
3. **User**: Represents a user account in the application. Contains personal information such as username, email, password, etc., validated using `userSchema`.
4. **App.js**: Entry point of the application, setting up middleware, loading environment variables from `.env`, and defining routes.
5. **Controllers**:
   - `listings.js`: Handles listing-related endpoints.
   - `reviews.js`: Handles review-related endpoints.
   - `users.js`: Handles user-related endpoints.
6. **Middleware**:
   - `isLoggedIn.js`: Middleware for checking authentication status.
   - `saveredirectUrl.js`: Middleware for saving and setting redirect URLs in locals.
7. **Models**:
   - `listing.js`: Represents the schema for listings.
   - `review.js`: Represents the schema for reviews.
   - `user.js`: Represents the schema for users.
8. **Utils**:
   - `ExpressError.js`: Utility file for creating custom Express errors.

## Database Schema

### Listing
```javascript
const listingSchema = new mongoose.Schema({
  title: { type: String, required: true },
  description: { type: String, required: true },
  location: { type: String, required: true },
  price: { type: Number, required: true, min: 0 },
  country: { type: String, required: true },
  images: [String]
}, {
  timestamps: true
});
```

### Review
```javascript
const reviewSchema = new mongoose.Schema({
  author: { type: mongoose.Schema.Types.ObjectId, ref: 'User' },
  listing: { type: mongoose.Schema.Types.ObjectId, ref: 'Listing' },
  content: { type: String, required: true }
}, {
  timestamps: true
});
```

### User
```javascript
const userSchema = new mongoose.Schema({
  username: { type: String, unique: true, required: true },
  email: { type: String, unique: true, required: true },
  password: { type: String, required: true }
}, {
  timestamps: true
});
```

These schemas are used by the models to interact with MongoDB.
